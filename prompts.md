## 提示词 2：方法论参考
```
好的，结合这份经验拆成多段提示词我即将开始搭建。

结合 CSDN、掘金、腾讯云社区等高质量技术论坛上的实战经验，以下从"核心秘诀"和"总体规划建议"两个维度，为你梳理一套高效、高质量、低"AI味"的网页开发方法论。

一、核心秘诀（5条）

秘诀 1：提示词不是写得长，而是写得"结构化"
很多人都以为给 AI 的提示词越长越好，但实际上关键不在长度，在于结构是否清晰。每条网页类提示词至少应该包含这5段信息：
- 页面目标：做什么页面，给谁看，用户要完成什么动作。
- 页面结构：必须包含哪些区块（如 Hero、卖点卡片、FAQ、CTA 等）。
- 视觉约束：明确技术栈、样式方案（如 Tailwind CSS）、配色、字体等，以及"不要什么"（见秘诀 4）。
- 修改范围：只允许修改哪些文件，是否允许新增依赖。
- 验收标准：给出手动验收步骤，覆盖桌面端、移动端和关键交互。

按照这个模板去组织提示词，AI 生成的代码会精准得多。比如：目标是新增一个 SaaS 落地页首屏，核心动作是引导用户点击"立即试用"。结构是 Hero、3 个卖点、客户评价、FAQ、页脚 CTA。风格是沿用当前站点色彩和按钮样式。范围是只允许修改 landing page 对应页面文件和一个新组件文件。验证是检查按钮、移动端适配、FAQ 折叠和文案层级。

实测数据显示，"总-分"结构的渐进式提示词比一次性长提示词的代码准确率高出约41%。

秘诀 2：先跑通 MVP，再迭代加功能
不要一上来就想做一个"完整的大网站"，网站建设正从"大而全"转向"小而美"。MVP（最小可行产品）的核心逻辑是先做一个"能跑通"的最精简版本：
- 只保留核心功能：用"用户价值 × 开发复杂度"矩阵筛选，只做"高价值 + 低复杂度"的功能；
- 先跑通再优化：让网站能"看起来像个完整的页面"并顺利运行，再在这个基础上逐渐添加细节功能；
- 代码版本管理：每改好一小块功能，立刻用 git 提交一次，这是犯错的"后悔药"。

秘诀 3：把复杂页面拆成独立模块，分步完成
分模块开发是高质量网页落地的关键。最实用的拆分策略是：
- 前端按"基础元素 → 组件 → 页面"三层拆分：原子元素（按钮、文本、输入框）→ 组件层（导航栏、轮播图、卡片列表）→ 页面层（首页、产品页）；
- 采用"整体布局 → 功能模块 → 交互细节"三阶段开发：第一阶段搭页面骨架（Container 组件 + 布局框架），第二阶段填充各功能模块，第三阶段实现动画、表单验证、响应式适配等细节；
- 模块间通过接口通信：每个模块只做"一类事"，互相之间不直接调用内部方法，降低耦合。

这样做的好处是：AI 在处理单一目标时精准度远高于处理混杂需求，拆分后输出精准度可提升80%以上。

秘诀 4：去"AI味"的核心——反向约束 + 细节干预
AI 生成的网页之所以一股"模板味"，根本原因是训练数据来源重叠（GitHub 开源项目、Tailwind/Bootstrap 示例代码等），导致输出偏好"蓝紫渐变"、"emoji 图标"、"柔和光晕"等套路。

去除 AI 味的核心技巧：
- 不说"做个好看的页面"，而是说"不要用 Inter 字体，不要紫色渐变，给我点独特的东西"；
- 明确禁止项：禁止大面积蓝紫渐变、禁止用 emoji 代替图标、禁止千篇一律的悬停上浮动画；
- 给 AI 一套可执行的设计规范，包含"该做什么"和"不该做什么"；
- 定制化素材：替换默认图标为开源可商用的图标库（如 Lucide、Heroicons），用真实的品牌图片替代占位图；
- 细节优化要"人工参与"：有些视觉和交互细节，AI 喂再多提示词也不如人直接上手改代码来得快。先让 AI 生成一份能跑通的版本，再在此基础上人工打磨细节。

秘诀 5：代码校验 + 规范化是质量保证
AI 生成的代码能跑不等于能维护。生成后必须做三件事：
- 接入 ESLint + Prettier 进行代码格式化，检查类型定义完整性、异步操作错误处理等；
- 删除 AI 生成的冗余代码：未使用的样式类、重复函数、多余变量等；
- 组件化重构：将生成的一整块页面代码按功能拆分为公共组件和页面组件，符合 Vue/React 组件化规范。

二、总体规划建议

基于以上秘诀，建议你按以下顺序组织项目规划：

第一阶段：需求定义 + 提示词准备（约占总时间 10%）
- 把网站需求拆成结构化提示词文档（包含目标、结构、视觉约束、范围、验收标准五要素）；
- 明确技术栈约束（如"Vue 3 + Tailwind CSS，禁止引入 jQuery"）；
- 制定设计规范约束（配色、字体、间距、禁止项清单）；
- 使用"总-分"结构组织提示词：先给出整体框架，再分模块细化。

第二阶段：MVP 快速落地（约占总时间 40%）
- 先用 AI 生成整体布局骨架（Header / Main Content / Footer 等大框架）；
- 再按模块逐个生成核心页面区块（导航栏、Hero 区、核心功能展示区、表单等）；
- 确保首版代码能在本地浏览器中正常运行、无明显报错；
- 初始化 Git 仓库，每完成一个模块就做一次 commit。

第三阶段：迭代优化 + 去 AI 味（约占总时间 35%）
- 加入 ESLint + Prettier，对 AI 生成代码进行格式化校验；
- 视觉去 AI 味：按设计规范替换图标库、调整配色、去除模板化动画效果；
- 文案去 AI 味：将 AI 生成的"智能化"、"赋能"等套话替换为有温度的真实文案；
- 交互打磨：补充细腻的动效反馈、修复响应式布局问题、完善逻辑闭环；
- 每轮改动后及时 git commit。

第四阶段：集成与部署（约占总时间 15%）
- 连接真实数据源（API 或 CMS），替换占位数据；
- 进行跨浏览器和移动端兼容性测试；
- 部署上线（可使用 GitHub Pages、Vercel、Netlify 等一键部署方案）。

推荐工具链：
- 需求梳理 + 产品定位：ChatGPT / Claude（充当"AI 产品经理"）
- 原型快速生成：v0.dev（一键生成可交互网页草稿）
- 代码编写与调试：Cursor（内置 AI 对话，支持规则配置和模块化开发）
- 图标素材：Lucide、Heroicons（开源可商用，替代 emoji 图标）
- 代码规范化：ESLint + Prettier
- 部署：Vercel / Netlify / GitHub Pages

几点重要提醒：
- 先有 0→1 的能力，AI 才能帮你实现 1→N：你至少要能判断 AI 产出的代码好不好、结构是否合理；
- 复杂页面一定要拆模块：不要让 AI 一次性生成一整页完整代码，按"组件"为颗粒度逐个生成，再组合；
- 随时做好版本回退准备：AI 改崩代码是常态，Git 是你最可靠的"后悔药"；
- 不要追求一步到位：先让网站能跑起来，再通过多轮对话和人工打磨逐步达到理想效果。
```

## 💡 开发经验总结

### 成功的经验
- 分模块开发，逐个突破
- 使用 Git 版本控制，保留历史版本
- 单文件版本便于分发，是一个很好的备份方案

### 踩过的坑
- JSON 数据加载会有跨域问题，需要服务器环境
- 印章位置调整了多次，最终选择了放在标题旁边
- 字体显示问题，特别是繁体书法字体的兼容性

### 下次可以改进的地方
- 提前考虑数据共享问题
- 更早引入响应式设计
- 多做用户测试
